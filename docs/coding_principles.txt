Coding principles
=================

Purpose
-------

This project exists to rebuild a personal Debian PHP/Laravel hosting server
quickly, calmly and repeatably from a blank machine.

It is intentionally opinionated. It is not trying to become a generic hosting
control panel, deployment platform or universal provisioning framework.

Prefer simple, boring and inspectable code over clever abstraction.


Scope
-----

The project is Debian-first and PHP-first.

The expected use case is personal hosting with a small number of sites, mostly
Laravel, administered by someone who is comfortable reading shell scripts and
server configuration files.

Interactive operation is the default. Non-interactive provisioning, import/export
systems and full control-panel behaviour should be added only if they become
clearly necessary.


Naming conventions
------------------

Function names use lowercase words separated by underscores.

Local variables use lowercase words separated by underscores.

Constants use uppercase words separated by underscores.

Script entrypoint functions should be namespaced to the script purpose. Do not
use a plain main function, because scripts may be read together or sourced during
maintenance.

Examples:

    main_core_health_check
    main_user_health_check
    main_website_health_check

Script-global mutable variables should be avoided. If they are justified, they
must be deliberately obvious:

    - uppercase
    - prefixed with MH_
    - narrowly scoped to the script's purpose

Examples:

    MH_WEBSITE_HC_PASS_COUNT
    MH_WEBSITE_HC_WARN_COUNT
    MH_WEBSITE_HC_FAIL_COUNT

Library functions should be namespaced with short, readable prefixes.

Preferred namespaces:

    log_
    prompt_
    svc_
    file_
    user_
    ssh_
    site_
    cert_
    fpm_
    mariadb_
    redis_
    apache_

Prefer readable 3-4 character prefixes where practical. Use a longer prefix when
the shorter form would be unclear.

Consistency matters more than perfect naming. Do not mix naming styles for the
same concept.


Variable scope
--------------

Use local variables inside functions wherever possible.

Avoid global variables unless they make the script materially simpler and easier
to maintain. Bash does not protect scope well, so accidental globals are a common
source of strange bugs.

If a global is used, make it obvious and intentional with the MH_ uppercase
convention described above.


Function design
---------------

Functions should do one clear thing.

A function may contain several OS calls if they are all part of one coherent
operation. For example, hardening one directory may involve chown and chmod.

Split functions when they start mixing responsibilities such as:

    - checking state
    - mutating state
    - logging or explaining
    - prompting
    - deciding whether failure is fatal
    - performing unrelated setup steps

Wrappers are acceptable when they are reused, centralise risky behaviour, or make
a repeated sequence safer and easier to maintain.

Avoid wrappers that merely hide control flow from the calling script.


Libraries and control scripts
-----------------------------

Shared libraries provide reusable facts and actions.

Libraries should not normally decide:

    - what is displayed to the user
    - whether a failure is fatal
    - whether an interactive repair should be offered
    - whether verbose output should be shown

Control scripts own:

    - orchestration
    - INFO/WARN/FAIL messages
    - prompts
    - retry or repair decisions
    - exit decisions

A library helper may answer a question:

    svc_is_active apache2

or perform one clear action:

    svc_reload apache2
    file_harden_server_admin_ssl_directory

The calling script decides the surrounding workflow.


Command output and logging
--------------------------

User-facing scripts should print concise status messages.

Raw stdout/stderr from noisy external commands should be captured in the script
log under /server-admin/logs rather than dumped to the terminal in normal mode.

Use the shared logging command helper for external commands whose output should
be captured consistently.

In normal mode, command output should normally go to the log file.

In verbose mode, command output may be shown to the user and also written to the
log file.

Libraries should not read global verbosity state or decide output policy. The
calling script chooses whether to run a command through the logging helper or to
capture/check output directly.

Avoid ad hoc redirection unless there is a clear reason.


Failure handling
----------------

Fail early on unsafe conditions.

A failure message should say:

    - what failed
    - what was not done
    - what to check next where practical

Avoid messages such as:

    Something went wrong.

Prefer messages such as:

    Apache configuration validation failed. Apache was not reloaded.

Do not continue after a failure if the next step could make the system harder to
recover.


Comments
--------

Comments should explain why something is being done, not merely restate what the
code already says.

Bad:

    # Restart Apache
    svc_restart apache2

Good:

    # Apache is shared by all hosted sites, so validate before reload to avoid
    # taking every site offline because of one bad generated vhost.
    apache_validate_config


Health checks
-------------

Health checks are diagnostic first.

Default mode should report only and should not change the system.

Interactive repair mode may offer low-risk repairs, but it must prompt before
changing anything.

Health checks should separate:

    - fact checks
    - presentation
    - repair actions

Dangerous changes belong in setup scripts, not silent health-check repair logic.


Configuration ownership
-----------------------

Prefer managed fragments over editing distribution-managed files directly.

For example, SSH policy should normally live in:

    /etc/ssh/sshd_config.d/server-admin.conf

rather than being repeatedly edited into:

    /etc/ssh/sshd_config

If a packaged file must be edited to enable managed fragments, the script should
back it up, explain what is happening and validate before reloading services.


Permissions and security
------------------------

Use least privilege.

The general model is:

    - root owns sensitive security boundaries
    - server_admin group maintains the toolkit
    - certificate private keys stay root-only
    - each website has its own Linux user
    - each website has its own PHP-FPM pool
    - each website should have its own database user
    - Apache reads public files and proxies to PHP-FPM
    - PHP-FPM runs application code as the site user

Avoid shared writable paths unless there is a specific reason.


Testing
-------

Local testing should normally use a disposable Debian WSL environment.

Target Debian versions should be explicit. Debian 13 is the current target until
a later Debian release has been tested and deliberately adopted.

Important changes should still be verified on a clean Debian server or VM before
being trusted for real hosting.
